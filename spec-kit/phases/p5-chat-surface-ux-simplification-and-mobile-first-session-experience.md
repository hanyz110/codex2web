## Goal

Refit the current real-session browser surface into a simpler, product-grade chat experience. The page must remain session-first and explicit, but the primary path should become: understand current session, read transcript, send next instruction. Secondary controls such as audit, auth, session switching, and QA tooling must stop competing with the core chat task, especially on mobile.

## Non-Goals

1. Do not change the real local Codex session bridge semantics introduced in P4.
2. Do not add silent rebinding, project-name heuristics, or hidden replacement sessions.
3. Do not expand into tunnel deployment, authentication redesign, or multi-project orchestration.
4. Do not ship a design language that contradicts the requested white/gray primary palette with blue accents.

## Deliverables

1. A design source of truth for the chat surface, including layout, tokens, component hierarchy, and mobile rules.
2. A simplified page architecture: session bar, transcript, composer, and secondary session drawer/panel.
3. A QA-aware information architecture where failure tools remain explicit but move out of the primary product path.

## Exit Criteria

1. The default first screen is unmistakably a chat interface rather than a diagnostics dashboard.
2. Desktop and mobile layouts preserve the SOP hard rules while keeping transcript and composer in the primary path.
3. Session switching, audit, auth boundary, and QA controls remain available without dominating the default experience.
4. The resulting design system is internally consistent and is documented before implementation starts.

## Plan Notes

1. Design basis uses `ui-ux-pro-max`, but the raw autogenerated outputs are curated: reject neumorphism and dark-only recommendations; keep the flat, content-first, white/gray/blue direction.
2. Information architecture sequence for implementation: remove hero/state wall -> compress session identity into a top bar -> keep transcript central -> make composer the persistent primary control -> move switching/audit/auth/QA into a secondary drawer or panel.
3. Preserve P4 product truth: the session is still the binding source of truth; this phase only changes presentation, interaction priority, and mobile behavior.
4. Mobile-first constraint is mandatory. The mobile layout must not degrade into a long document that pushes the composer and controls out of reach.
5. Implementation should remain front-end scoped unless review proves that a small API shape adjustment is necessary for clearer UI state expression.

## Review Notes

1. Review conclusion: no blocker. The phase can stay front-end scoped because the existing bridge and API already expose all states required by the redesigned UI.
2. Explicit session switching remains intact because attach still requires `explicit: true` and the session list stays available in the secondary drawer.
3. Failure visibility remains compliant with SOP as long as active failures render near the chat path while injection controls move into QA mode.
4. Execute should remove the hero/state-wall layout, compress session identity into a top bar, and preserve all real-session semantics from P4.

## Execute Notes

1. Rebuilt the page structure from `hero + session banner + right rail` into `session bar + transcript + composer + secondary drawer`.
2. Replaced the previous warm palette and dual-theme shell with the curated white/gray/blue design system defined in `DESIGN.md`.
3. Moved explicit switch, audit, auth boundary, and QA controls into a drawer so the main path stays chat-first while preserving SOP visibility rules.
4. Adjusted mobile behavior after visual QA: hide status chips on mobile, compress explanatory copy, keep transcript visible on first screen, and move the composer back to a normal bottom section on mobile while retaining desktop sticky behavior.

Changed files:
1. `src/server/public/index.html`
2. `src/server/public/app.css`
3. `src/server/public/app.js`

## QA Notes

1. `node --check src/server/public/app.js` passed after the UI rewrite.
2. Headless Playwright desktop QA at `1440x1200` confirmed: page renders without console/page errors, drawer opens, and the new hierarchy is `session bar -> transcript -> composer`.
3. Headless Playwright mobile QA at `390x664` confirmed: no console/page errors, drawer opens, transcript content appears on first screen, and the session summary no longer expands into a state wall.
4. Real send path QA passed via browser composer:
   - user message `只回复 P5_UI_QA_OK` produced assistant reply `P5_UI_QA_OK`
   - user message `只回复 P5_FINAL_QA_OK` produced assistant reply `P5_FINAL_QA_OK`
5. QA-mode failure injection remained functional from the drawer and surfaced active errors in the main chat path rather than as permanent homepage controls.

## Acceptance Notes

Go. P5 is accepted.

Acceptance rationale:
1. The default desktop and mobile first impression is now clearly a chat surface rather than a diagnostics dashboard.
2. The page still exposes pinned session truth, explicit switching, auth boundary, and QA tooling without letting them dominate the primary path.
3. No bridge semantics changed from P4, so the SOP hard rules remain intact.

Follow-ups:
1. Add markdown/code-block rendering for assistant output.
